# Deploy Notes API on Render

This guide uses the included **Dockerfile** and **render.yaml** Blueprint.

## Files included

| File | Purpose |
|------|---------|
| `Dockerfile` | Multi-stage build (Maven 21 → JRE 21 JAR) |
| `render.yaml` | Blueprint: web service + free PostgreSQL |
| `.dockerignore` | Keeps Docker build fast and small |
| `.env.render.example` | Env vars to set manually in Render UI |
| `RenderDatabaseEnvironmentPostProcessor` | Converts `postgresql://` → JDBC for Spring |

## Prerequisites

1. GitHub (or GitLab) repo with this project pushed
2. [Render](https://render.com) account

## Option A — Blueprint (recommended)

1. Push code to GitHub.
2. Render Dashboard → **New** → **Blueprint**.
3. Connect the repository; Render detects `render.yaml`.
4. Review resources:
   - **notes-api** — Docker web service (free)
   - **notes-db** — PostgreSQL (free)
5. Set **manual** env vars when prompted (or after deploy):
   - `ABOUT_NAME` — your name
   - `ABOUT_EMAIL` — your email  
   (`JWT_SECRET` is auto-generated by the Blueprint.)
6. Click **Apply** / **Deploy**.

Render will build the Docker image, provision Postgres, and inject `DATABASE_URL`.

## Option B — Manual Web Service

1. **New** → **Web Service** → connect repo.
2. **Language:** Docker (Render uses root `Dockerfile`).
3. **Plan:** Free.
4. **New** → **PostgreSQL** → create `notes-db`.
5. On the web service → **Environment**:
   - Link database (add `DATABASE_URL` from Postgres **Internal** connection string).
   - Add variables from `.env.render.example`.
   - Set `SPRING_PROFILES_ACTIVE=postgres`.
   - Set `JWT_SECRET` to a random string (32+ characters).
6. **Health Check Path:** `/about`
7. Deploy.

## Build & start (handled automatically)

- **Docker build:** `mvn clean package` inside Dockerfile.
- **Start:** `java -jar app.jar` (listens on `$PORT` from Render).

No custom build/start commands needed when using Docker.

## Verify deployment

Replace `YOUR-SERVICE` with your Render URL (e.g. `notes-api-xxxx.onrender.com`):

```bash
curl https://YOUR-SERVICE.onrender.com/about
curl https://YOUR-SERVICE.onrender.com/openapi.json

curl -X POST https://YOUR-SERVICE.onrender.com/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"password123"}'
```

## Troubleshooting

| Issue | Fix |
|-------|-----|
| **Build failed** (Docker) | Open **notes-api → Logs → Build**. Common fix: redeploy after pulling latest `Dockerfile` (removed broken `mvn go-offline` step). |
| **Deploy failed** / health check | Open **Logs → Deploy**. Wait up to 5 min on free tier. Ensure `DATABASE_URL` is set (link **notes-db**). |
| `Driver claims to not accept jdbcUrl, postgresql://...` | Redeploy with latest code — `RenderDatabaseEnvironmentPostProcessor` converts URL to JDBC. |
| `Connection to localhost:5432 refused` | **Database not linked** to notes-api. See [Link the database](#link-the-database-critical) below. |
| `The connection attempt failed` | **Delete `DATABASE_URL`** (external). Use internal `DB_HOST` (`dpg-xxxxx-a`). See below. |
| DB connection error | Confirm **notes-db** is available; `SPRING_PROFILES_ACTIVE=postgres`; `DB_HOST` or `DATABASE_URL` set in Environment. |

### Fix "The connection attempt failed"

Render's `DATABASE_URL` / `connectionString` is often the **external** hostname (`....oregon-postgres.render.com`), which can fail from inside your Docker container.

1. **notes-api → Environment** → delete **`DATABASE_URL`** if it exists
2. Keep only internal fields from **notes-db**: `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`
3. `DB_HOST` must look like `dpg-xxxxx-a` (short internal name), not `....render.com`
4. Redeploy with latest code (Blueprint no longer injects `DATABASE_URL`)

Deploy logs should show: `Using internal PostgreSQL at dpg-xxxxx-a:5432/notes_db`

### Link the database (critical)

If logs show `localhost:5432`, Render is **not** passing database credentials to the web service.

1. Render Dashboard → **notes-api** → **Environment**
2. Confirm these exist (values hidden): `DATABASE_URL`, `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`
3. If missing: **Add Environment Variable** → **Add from database** → select **notes-db** → add connection string and/or individual fields
4. **Save** and **Manual Deploy**

Or redeploy the whole Blueprint so `render.yaml` injects them automatically.
| 502 on cold start | Free tier sleeps ~50s; retry or upgrade plan |
| Invalid JWT / startup error | Set `JWT_SECRET` to 32+ characters in Environment (or use Blueprint `generateValue`) |
| Data lost after deploy | Do not use SQLite on Render; use linked Postgres |

### Read the failure log

1. Render Dashboard → **notes-api** → **Logs**
2. If **Build** tab shows errors → Docker/Maven issue (fix Dockerfile, push, redeploy)
3. If **Deploy** tab shows `Application failed to start` → expand stack trace:
   - `postgresql://` in JDBC error → pull latest code and redeploy
   - `JWT_SECRET must be at least 32` → add/regenerate `JWT_SECRET` in Environment
   - `Connection refused` / `timeout` → database not linked or still provisioning; redeploy after **notes-db** is live

## Notes

- Free web services spin down after inactivity; first request may be slow.
- Use Postgres **internal** URL when app and DB are on Render (already done in Blueprint).
- Hibernate `ddl-auto=update` creates tables on first run.
